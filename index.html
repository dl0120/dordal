2 Introduction

  • chapter 1 is a summary of "essentials":
    • LAN
    • IP
    • TCP
    • datagram forwarding
    • packets
    • congestion
    • sliding windows
    • firewalls
    • network address translation
  • "LAN"
    • "physical" networks that provide the "connection" between different machines
  • "local"
  • "IP"
    • a "layer"
    • connects many LANs into the Internet
  • "TCP"
    • related to "transport", "connections", sending user data
  • "datagram forwarding"

3 1.1. Layers

  • "layers"
    • the entire topic of [starting with some data] and [sending it out] is complicated
    • so, divide the complicated topic into different parts
    • a "layer" represents one part
    • one feature/function/aspect
  • "four-layer model"
    • (1) Application layer: creates some data
    • (2) TCP layer
    • (3) IP layer
    • (4) LAN layer: actually delivers
  • 5-layer model
    • divide LAN layer into "physical" LAN layer and "logical" LAN layer
  • 7-layer model

4 1.2. Data Rate, Throughput and Bandwidth

  • "data rate"
    • the rate at which bits are transmitted
  • "Throughput"
    • the actual overall rate at which bits are transmitted
    • things like competing traffic can affect the "Throughput"
  • "bandwidth"
    • can mean either "data rate" or "throughput"
    • we use it mostly to mean "data rate"
  • "goodput"
    • the actual rate at which bits were received by destination application
  • kilobit
    • 103 bits
  • megabit
    • 106 bits
  • kbps and Mbps v. kB and MB
    • kB and MB denote data "VOLUMES" of 210 and 220 bytes
    • the upper-case B denotes bytes
    • newer abbreviation: KiB and MiB

5 1.3. Packets

  • "packet"
    • a unit of data that's transferred
    • has header
  • "header"
    • meta-data portion of a packet
    • fex: datagram forwarding: meta-data includes destination address
    • fex: virtual-circuit forwarding: meta-data includes identifier for the connection
  • Single and multiple headers.png
  • "frames", "segments"
    • frames: packet at the LAN layer
    • segments: packet at the Transport layer
  • packet size is limited
    • the type of LAN determines max size of a packet
    • ethernet: 1500 bytes
  • one issue: how do I send a packet from a [large-packet LAN] to a [small-packet LAN]?
    • IP layer addresses this
  • each layer adds its own header
    • each layer adds its own header
    • each header is about ~20 bytes
    • so, in a 512-bytes-packet, it's not insignificant
  • "datagram forwarding network" and headers
    • the header will contain [destination address] and other delivery info]
    • "routers" or "switches" try to ensure that the packet is delivered to the requested destination
  • packets and bytes
    • today, everyone agrees on 8-bit bytes
    • in the past, this wasn't always so
    • there's still some ambiguity:
    • "binary integers" can be represented as a sequence of bytes in either "big-endian" or "little-endian" "byte order"
    • RFC 1700: IP should use big-endian byte order
    • so, big-endian byte order is aka "network byte order"

6 1.4. Datagram forwarding

  • "datagram-forwarding"
    • one way of delivering packets
    • [packet headers] have a "destination address"
    • "routers"/"switches" are to read this to send the packet to correct address
  • "forwarding table"
    • maps a [destination address] and [nexthop]
  • "nexthop"
    • the immediate "neighbor address" or "interface" toward the destination
  • "switch", "interface", and "neighbor"
  • network diagram without a loop.png
  • "stateless forwarding"
    • forwarding tables don't have a "per-connection state"
  • [datagram forwarding] sometimes has other info in the header,
    • aside from destination address
    • "quality of service information"
    • there's a TON to transfer: use path A
    • I need this transferred quick: use path B
  • "switches", "bridges" v. "routers"
    • switches/bridges work on the LAN layer
    • routers work on the IP layer
  • "default entry" in a forwarding table
    • where to send a packet, if no match exists on the forwarding table
  • aside: virtual circuits, state-ful forwarding
    • different connections can be routed differently

7 1.5. Toplogy

  • network without a "loop"
    • there's only ONE path between any pair of nodes
    • aka "acyclic", "tree"
    • there's no "redundancy"
  • network with a "loop"
    • a pair of nodes has more than 1 path

8 1.5.1. Traffic engineering

  • "traffic engineering"
    • when there's redundant paths between 2 nodes,
    • intentionally choosing 1 route over another
  • "quality of service information" in datagram forwarding
    • can do "traffic engineering"

9 1.6. Routing loops

  • "routing loop"
    • in a network with a loop, a packet might circulate endlessly
  • forwarding table is "distributed"
    • there is no "global authority" that does sanity checks on different forwarding tables
  • "linear routing loop"
    • a "routing loop" can happen in a network WITHOUT a loop too!
    • 2 nodes just send a packet back and forth infinitely
  • how to avoid routing loop?
    • maybe disallow a loop in the network "topology" altogether
    • maybe forbid a switch send a packet "back out" the path ("interface") by which it arrived
    • maybe have a "time to live" - a packet can only hop so many times before it is destroyed
  • "time to live" field, "TTL field"
  • node D might decide that the best path to B is D-E-C-B
    • node E might decide that the best path to B is E-D-A-B
    • D and E would send the packet back and forth to each other
    • a linear loop arises
  • successful datagram routing requires cooperation (between nodes) and a consistent view of the network (held by different nodes)

10 1.7. Congestion

  • "congestion"
    • packets arrive faster than they can be sent out
    • bottleneck, backlog
  • queue
    • when there's a backlog,
    • a "queue" forms
    • when the queue is full, packets will be "dropped"
    • basically, ignore new packets
  • "knee"
    • the point where a queue is beginning to buidl up
    • aka "contention"
  • "cliff"
    • the point where queue is full, and packets are being dropped
  • congestion doesn't necessarily mean "not enough bandwidth"
    • something about "network's feedback that the maximum transmission rate has been reached"
